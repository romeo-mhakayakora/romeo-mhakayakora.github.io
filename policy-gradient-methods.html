Module 02 // Course Notes

Policy Gradient Methods

Comparing action-value and policy parameterizations, deriving the log-derivative trick, and stabilizing Monte Carlo estimators with state-value baselines.

Setting the Stage: Two Families of RL Methods

Before diving into REINFORCE, it's worth placing policy gradient methods in context — because they represent a fundamental shift in what we're trying to learn.

1. Action-value methods (what you've likely seen so far — Q-learning, SARSA, DQN): here, the policy doesn't really exist on its own. We learn $Q(s,a)$ — how good each action is — and the policy is derived from it, usually as $\pi(s) = \arg\max_a Q(s,a)$. Remove the Q-values, and the agent has nothing to act on. The policy is a side effect of value estimation.

2. Policy gradient methods (what we're covering now): here we skip the middleman. Instead of learning values and deriving a policy, we parameterize the policy itself — $\pi_\theta(a|s)$ — and adjust $\theta$ directly, via gradient ascent, to make it perform better:

$$\theta_{t+1} = \theta_t + \alpha \nabla_\theta J(\theta)$$

A value function might still show up (as we'll see with baselines and actor-critic), but only as a helper to make learning more stable — it's optional, not a prerequisite for choosing actions.

Why this shift matters:

  • Action-value methods ask "what is this state/action worth?" and derive behavior from that.
  • Policy gradient methods ask "what should I do?" directly — the policy is the primary object being optimized, not a byproduct.

This is what lets policy gradient methods learn stochastic optimal policies (like a 33/33/33 mix in rock-paper-scissors, which value-based greedy selection can never represent), and lets the policy be any differentiable function — linear, neural network, Gaussian — as long as $\nabla_\theta \pi(a|s,\theta)$ exists.

REINFORCE — Monte Carlo Policy Gradient

From the Policy Gradient Theorem to an algorithm you can actually run, using only the returns the agent experiences.

00 — The Core Strategy: Sample, Don't Compute

We want to climb the performance surface $J(\theta)$ via gradient ascent:

$$\theta_{t+1} = \theta_t + \alpha \nabla J(\theta)$$

The problem: we can't compute $\nabla J(\theta)$ exactly — it depends on the environment's dynamics, which we don't know.

The critical insight: we don't need the exact gradient. We only need samples whose expectation equals the true gradient. Any proportionality constant just gets absorbed into $\alpha$.

The Policy Gradient Theorem gives us the exact (but uncomputable) form:

$$\nabla J(\theta) \propto \sum_s \mu(s) \sum_a q_\pi(s,a)\, \nabla_\theta \pi(a|s,\theta)$$
  • $\mu(s)$ — how often we visit state $s$ under $\pi$ (unknown)
  • $q_\pi(s,a)$ — true action-value (unknown)
  • $\nabla_\theta \pi(a|s,\theta)$ — the direction that increases the probability of $a$

We can't sum over all states, we don't know $\mu(s)$, and we don't know $q_\pi$. So we need to turn this into something we can sample by just playing episodes.

01 — First Transformation: Sum → Expectation

If we follow $\pi$, we naturally visit states in proportion to $\mu(s)$. So the outer sum is just an expectation in disguise:

$$\sum_s \mu(s)[\ldots] \;\longrightarrow\; \mathbb{E}_\pi\left[\sum_a q_\pi(S_t,a)\,\nabla_\theta \pi(a|S_t,\theta)\right]$$

What just happened: we traded an impossible sum over all states for an expectation we can sample — drop the agent in the environment, let it act, and the states it visits show up with exactly the right frequency automatically. The inner sum over actions is our next target.

02 — The Fork: All-Actions vs. REINFORCE

At this point there are two ways forward:

All-Actions Method REINFORCE
Mechanism Sum over all actions using a learned $\hat q(s,a,w)$ Use only $A_t$, the one action actually taken
Information Dense — every action's estimated value Sparse — one sample, but exact
Bias Biased if $\hat q \neq q_\pi$ Unbiased — uses the true observed return $G_t$
Cost Heavy — gradient for every action Light — one gradient per step

REINFORCE avoids learning a separate value function entirely. It uses the actual return $G_t$ as an unbiased stand-in for $q_\pi(S_t,A_t)$. To get there, we need to eliminate the $\sum_a$ — that's the log-derivative trick.

03 — The Trick: Collapsing the Sum to One Sample

We want: $\sum_a q_\pi(S_t,a)\nabla_\theta \pi(a|S_t,\theta) \;\rightarrow\;$ a single-sample expectation.

Step 1 — Multiply and divide by $\pi(a|S_t,\theta)$ (multiplying by 1, legal at any time):

$$\sum_a q_\pi(S_t,a)\nabla_\theta\pi(a|S_t,\theta) = \sum_a \pi(a|S_t,\theta)\left[q_\pi(S_t,a)\,\frac{\nabla_\theta\pi(a|S_t,\theta)}{\pi(a|S_t,\theta)}\right]$$

Now $\pi(a|S_t,\theta)$ sits where a probability weight belongs — exactly the shape of an expectation.

Step 2 — Apply the log-derivative identity. Since $\nabla_\theta \ln x = \frac{\nabla_\theta x}{x}$:

$$\frac{\nabla_\theta \pi(a|S_t,\theta)}{\pi(a|S_t,\theta)} = \nabla_\theta \ln \pi(a|S_t,\theta)$$

Step 3 — Recognize the expectation. The sum is now $\sum_a \pi(a|S_t,\theta)\big[q_\pi(S_t,a)\nabla_\theta \ln\pi(a|S_t,\theta)\big]$, which is exactly $\sum P(x)f(x) = \mathbb{E}[f(x)]$:

$$\mathbb{E}_{A_t\sim\pi}\Big[q_\pi(S_t,A_t)\,\nabla_\theta \ln \pi(A_t|S_t,\theta)\Big]$$

Step 4 — Swap $q_\pi$ for the sampled return $G_t$. Since $\mathbb{E}_\pi[G_t \mid S_t,A_t] = q_\pi(S_t,A_t)$:

$$\nabla J(\theta) \propto \mathbb{E}_\pi\Big[G_t\,\nabla_\theta \ln \pi(A_t|S_t,\theta)\Big]$$

The pattern to remember: every "$\sum P(x)[\ldots]$" became "$\mathbb{E}[\ldots]$". We manufactured a probability weight out of thin air ($\times\pi/\pi$), pattern-matched to the expectation definition, and finally replaced the unknown $q_\pi$ with the one number we can actually observe — the return.

04 — The Algorithm

$$\theta_{t+1} = \theta_t + \alpha\, G_t\, \nabla_\theta \ln \pi(A_t|S_t,\theta_t)$$ 
REINFORCE: Monte-Carlo Policy-Gradient (episodic)

Input: differentiable policy π(a|s,θ)
Parameter: step size α > 0
Initialize θ (e.g. to 0)

Loop forever (each episode):
    Generate an episode S0,A0,R1,...,S_{T-1},A_{T-1},R_T following π(·|·,θ)
    Loop for t = 0, 1, ..., T-1:
        G ← Σ_{k=t+1}^{T} γ^{k-t-1} R_k        # return from t onward
        θ ← θ + α γ^t G ∇ln π(A_t|S_t, θ)

The $\gamma^t$ factor appears in the boxed algorithm for the general discounted case (the derivation above assumed $\gamma=1$ for clarity) — it down-weights updates for steps further from the episode's start.

05 — Deconstructing the Update

$$\theta_{t+1} = \theta_t + \underbrace{\alpha}_{\text{step size}} \cdot \underbrace{G_t}_{\text{"gas pedal"}} \cdot \underbrace{\nabla_\theta \ln \pi(A_t|S_t,\theta)}_{\text{"steering wheel"}}$$
  • $\nabla_\theta \ln \pi(A_t|S_t,\theta)$ — the steering wheel. Points in parameter space toward higher probability of repeating $A_t$. It doesn't know or care whether $A_t$ was good — only how to make it more likely.
  • $G_t$ — the gas pedal. Scales how hard we turn the wheel. Big positive return → floor it. Small or negative return → ease off, or reverse.

This is the whole mechanism: try something, see the return, steer probability mass toward it in proportion to how good it was.

06 — The Problem: Variance

$G_t$ is a full Monte Carlo sum of every reward from $t$ to the end of the episode. Two issues:

  • It's noisy — a single lucky or unlucky episode swings $G_t$ wildly, even for the exact same state-action pair.
  • In many environments it's always positive (or always negative), which means the gas pedal is always pushed forward — every action gets reinforced to some degree, just by different amounts. The algorithm can't cleanly distinguish "great" from "just okay."

This motivates the fix below.

REINFORCE with Baseline

Fixing the variance problem without changing the expectation — subtracting a learned baseline makes Monte Carlo policy gradient learn dramatically faster.

00 — The Motivation

🎰 The Slot-Machine Problem
Imagine an agent whose rewards are always between +90 and +100. Every return is positive, so every action looks "good," and the agent keeps reinforcing whatever it happened to try — it can't tell "great" (+100) from "mediocre" (+90). The resulting gradient is noisy: the same state-action pair can get wildly different updates purely from randomness in the episode.

The core insight: we don't care about the absolute return — we care about the relative return. Was this action better or worse than what we'd normally expect from this state? We need to center the learning signal around zero, not let it float at an arbitrary positive (or negative) baseline.

The fix: use $G_t - b(S_t)$ instead of $G_t$, where $b(S_t)$ is some function of the state only.

01 — The Generalized Theorem

$$\nabla J(\theta) \propto \sum_s \mu(s) \sum_a \big[q_\pi(s,a) - b(s)\big]\, \nabla_\theta \pi(a|s,\theta)$$

where $b(s)$ can be anything — random, learned, even zero — with one constraint: it must depend only on $s$, never on $a$.

Proof that subtracting $b(s)$ changes nothing in expectation:

$$\sum_a b(s)\,\nabla_\theta \pi(a|s,\theta) = b(s)\underbrace{\sum_a \nabla_\theta \pi(a|s,\theta)}_{=\;\nabla_\theta \sum_a \pi(a|s,\theta) \;=\; \nabla_\theta 1 \;=\; 0} = 0$$

Since probabilities over actions always sum to 1, their gradients sum to zero — so we're subtracting exactly nothing. The expected update direction is unchanged; only the variance of our sample changes.

Why does it have to be $b(s)$ and not $b(s,a)$? If the baseline depended on the action, we couldn't pull it out front of the $\sum_a$ — and the cancellation above wouldn't hold. That's exactly why we use a state-value function $v(s)$ as the baseline, never an action-value function $q(s,a)$.

Intuition for the zero-sum: $\nabla_\theta \pi(a|s,\theta)$ is "how to raise the probability of $a$." Because probabilities must sum to 1, raising one action's probability necessarily lowers another's — the increases and decreases already balance. Shifting every $q$-value by the same constant $b(s)$ preserves that balance; only the relative differences between actions (which is what actually drives learning) survive.

02 — What Should the Baseline Be?

The natural choice that minimizes variance is the true state-value function:

$$b(s) = v_\pi(s) = \mathbb{E}_\pi[G_t \mid S_t = s]$$

This turns $G_t - v_\pi(s)$ into something meaningful — an estimate of advantage: how much better this particular trajectory did compared to what's typical from this state.

Since we don't know $v_\pi(s)$, we learn it with a function approximator $\hat v(s,w)$, trained alongside the policy. This creates two parallel learners:

Actor (policy) Critic (baseline)
Parameters $\theta$ $w$
Goal Maximize expected return Predict expected return from each state
Update $\theta \leftarrow \theta + \alpha_\theta\,\delta\,\nabla\ln\pi(A_t\vert S_t,\theta)$ $w \leftarrow w + \alpha_w\,\delta\,\nabla \hat v(S_t,w)$
Role of $\delta$ Scales the policy gradient Supervised regression error

Not Actor-Critic yet. True Actor-Critic bootstraps — the critic's target is $R_{t+1} + \gamma \hat v(S_{t+1},w)$. Here we still use the full Monte Carlo return $G_t$ as the target. This is supervised regression on complete episodes, not TD learning — the critic never guides the policy mid-episode, it only reduces variance after the fact.

03 — Learning the Baseline: Plain Supervised Regression

We treat this as ordinary regression: "labels" are the observed returns $G_t$, "predictions" are $\hat v(S_t,w)$.

Loss: $$L(w) = \tfrac{1}{2}\big(G_t - \hat v(S_t,w)\big)^2 = \tfrac{1}{2}\delta^2$$

Gradient: $$L(w) = -\big(G_t - \hat v(S_t,w)\big)\nabla_w \hat v(S_t,w) = -\delta\,\nabla_w \hat v(S_t,w)$$

Descent step (move opposite the gradient): $$w \leftarrow w - \alpha_w(-\delta\,\nabla_w\hat v) = w + \alpha_w\,\delta\,\nabla_w \hat v(S_t,w)$$

The shared $\delta$: the same error term drives both updates, but for two different purposes:

  • for $w$ → "pull my prediction closer to $G_t$"
  • for $\theta$ → "scale my policy update by how surprising $G_t$ was"

Even a trivial baseline helps: in Sutton & Barto's short-corridor example, all states look identical, so $\hat v(s,w)=w$ is just a single scalar — yet it's enough to separate "less bad" from "more bad" when every return is negative, which raw REINFORCE can't do at all.

04 — The Algorithm

REINFORCE with Baseline (episodic)

Input: differentiable policy π(a|s,θ)
Input: differentiable state-value function v̂(s,w)
Parameters: step sizes α_θ > 0, α_w > 0
Initialize θ, w (e.g. to 0)

Loop forever (each episode):
    Generate an episode S0,A0,R1,...,S_{T-1},A_{T-1},R_T following π(·|·,θ)
    Loop for t = 0, 1, ..., T-1:
        G ← Σ_{k=t+1}^{T} γ^{k-t-1} R_k        # actual return from t
        δ ← G − v̂(S_t, w)                       # advantage estimate
        w ← w + α_w · δ · ∇v̂(S_t, w)             # critic: regression step
        θ ← θ + α_θ · γ^t · δ · ∇ln π(A_t|S_t,θ)  # actor: policy step

Note $\delta$ replaces $G_t$ everywhere in the policy update — that's the entire change from vanilla REINFORCE.

05 — Vanilla REINFORCE vs. REINFORCE with Baseline

REINFORCE REINFORCE + Baseline
Signal used $G_t$ (raw return) $G_t - \hat v(S_t,w)$ (advantage)
Question asked "Was the return big?" "Was the return bigger than expected?"
Bias Unbiased Still unbiased (proven above)
Variance High Substantially lower
Extra machinery None A learned $\hat v(s,w)$, trained by regression
Still Monte Carlo? Yes Yes — no bootstrapping, full episode needed
