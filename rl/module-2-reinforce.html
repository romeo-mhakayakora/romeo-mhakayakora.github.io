Algorithm

REINFORCE is an on-policy algorithm (see On-Policy note below).

Setup:

  • Initialize learning rate $\alpha$.
  • Construct a policy network $\pi_\theta$ with randomly initialized weights $\theta$.

Per episode:

  1. Use $\pi_\theta$ to generate a trajectory $\tau = s_0, a_0, r_0, \ldots, s_T, a_T, r_T$.
  2. For each time step $t$ in the trajectory, compute the return $R_t(\tau)$.
  3. Use $R_t(\tau)$ to estimate the policy gradient at each time step, and sum the policy gradients across all time steps.
  4. Use the summed result to update the policy network parameters $\theta$.

Algorithm 2 — REINFORCE algorithm (pseudocode)

Algorithm 2: REINFORCE
1: Initialize learning rate $\alpha$ 2: Initialize weights $\theta$ of a policy network $\pi_\theta$ 3: for episode = 0, ..., MAX_EPISODE do 4: Sample a trajectory $\tau = s_0, a_0, r_0, \ldots, s_T, a_T, r_T$ 5: Set $\nabla_\theta J(\pi_\theta) = 0$ 6: for $t = 0, \ldots, T$ do 7: $R_t(\tau) = \sum_{t'=t}^{T} \gamma^{t'-t} r_{t'}$ 8: $\nabla_\theta J(\pi_\theta) \mathrel{+}= R_t(\tau) \nabla_\theta \log \pi_\theta(a_t \mid s_t)$ 9: end for 10: $\theta = \theta + \alpha \nabla_\theta J(\pi_\theta)$ 11: end for

On-policy nature of REINFORCE

Key point: A trajectory must be discarded after each parameter update — it cannot be reused.

Reason: REINFORCE is an on-policy algorithm — the parameter update equation depends on the current policy.

Evidence in Line 8: the policy gradient directly depends on action probabilities $\pi_\theta(a_t\mid s_t)$ generated by the current policy $\pi_\theta$ — not some past policy $\pi_{\theta'}$. Correspondingly, the return $R_t(\tau)$ (where $\tau \sim \pi_\theta$) must also be generated from the current $\pi_\theta$ — otherwise action probabilities would get adjusted based on returns they didn't actually generate.

Improving REINFORCE

The variance problem

REINFORCE estimates the policy gradient using Monte Carlo sampling with a single trajectory. This estimate is unbiased, but has high variance — returns can vary significantly from trajectory to trajectory.

Three sources of this variance:

  1. Actions have randomness (sampled from a probability distribution).
  2. The starting state may vary per episode.
  3. The environment's transition function may be stochastic.

Baseline (variance reduction)

One way to reduce variance: modify the returns by subtracting a suitable action-independent baseline $b(s_t)$:

$$\nabla_\theta J(\pi_\theta) \approx \sum_{t=0}^{T} \big(R_t(\tau) - b(s_t)\big)\nabla_\theta \log \pi_\theta(a_t\mid s_t) \tag{1.24}$$

Option 1 — Value function baseline: $b = V^\pi$ (the value function). This choice motivates the Actor-Critic algorithm (covered separately).

Option 2 — Mean return baseline:

$$b = \frac{1}{T}\sum_{t=0}^{T} R_t(\tau)$$

  • This is a constant baseline per trajectory — it does not vary with state $s_t$.
  • Effect: centers the returns for each trajectory around 0.
  • Result: for each trajectory, the best 50% of actions get encouraged, and the rest get discouraged, on average.

Why this matters — the all-negative/all-positive reward case

All-negative-reward environments: without a baseline, even a very good action gets discouraged, because returns are always negative.

  • Over time this can still yield good policies, since worse actions get discouraged even more — indirectly increasing the relative probability of better actions.
  • Downside: slower learning, since probability adjustments can only be made in a single direction (down).

All-positive-reward environments: the converse problem happens.

More effective approach: directly both increase and decrease action probabilities — this requires having both positive and negative returns, which is exactly what the baseline (via mean-centering) achieves.
